# 语迹 VoiceLog · 账号、多端同步与数据存储设计（规划草案 v0.1）

> 状态：**设计草案，待与产品对齐后再开发**。不影响现有原型运行。
> 关联：PRD「六、技术架构」「八、账号与多端同步」。
> 起因：现有原型数据只存浏览器 localStorage，且网页端/手机端各存各的、互不相通；
> 需要规划「多端同步 + 数据存储」，并新增「微信登录 / 手机号登录」。

---

## 0. 现状（要先认清的地基问题）

- 存储：纯 `localStorage`，在**当前设备、当前浏览器**里。无云端、无后端。
- 分裂：网页端 key = `voicelog-web-v2`，移动端 key = `voicelog:<主题>`（还按主题分裂）。
  → **同一台电脑、同一个浏览器里，手机界面和网页界面的数据都不互通。**
- 成长值（XP / 等级 / 累计天数）是**独立存储的可变状态**，不是从事件派生——这会让同步容易重复计数。
- 语音识别只实现了浏览器 Web Speech 兜底，且该 API 依赖 Google 后端、国内不可用 → 实际回退示例。

---

## 1. 必须守住的原则（来自 PRD 第四节）

1. **无账号也能完整用**——同步/登录是「增强」，不是「前提」（赋能不强制）。
2. **离线随时可写**，联网后自动合并。
3. **数据本地永远完整、可一键导出**（不锁定）。
4. **冲突温和处理、绝不静默丢数据**（抹平摩擦）。

> 推论：登录的定位是「**解锁多端同步与跨设备恢复**」，而不是「使用产品的门槛」。

## 2. 数据存储设计

### 2.1 统一存储层（Store 适配器）—— 同步前的必做地基
把现在散落的 `localStorage.setItem/getItem` 收敛成一个抽象接口：

```
Store {
  read(collection)            // 读全部 / 按天
  write(record)               // 落库（带 id / updatedAt）
  subscribe(fn)               // 变更订阅，驱动 UI
  exportAll() / importAll()   // 本地优先：随时导出/导入
}
```

- 底层实现：**IndexedDB**（替代 localStorage：容量大、可存长期历史、支持索引）。
  可借 Dexie 简化。
- **网页端与移动端共用同一份 Store**，彻底消除「两端各存各的」。
- 现有原型只需把直接读写 localStorage 的地方换成调用 Store；UI 不动。

### 2.2 数据模型（同步友好）
- 每条记录（日程 / 备注 / 回收站项）统一带：
  `id`（全局唯一，建议 UUID）、`updatedAt`（逻辑时钟/时间戳）、`deleted`（墓碑位）、`rev`/`deviceId`。
- **删除用 tombstone（墓碑式软删）**，不真删——这正好**复用语迹已有的「回收站」**：
  删除 = 打标记并同步出去，避免「A 删了、B 又复活」，回收站还能跨端找回。
- **成长值改为「可派生」**：XP / 等级 / 累计天数尽量**从事件流重新算**，而非独立存。
  这样同步只需同步「事件」，成长值自然一致，也天然守住「只升不降」。
  （过渡期若保留独立值，同步时一律**取两端最大**。）

## 3. 多端同步设计

### 3.1 路线对比

| 路线 | 机制 | 优点 | 代价 |
|---|---|---|---|
| ① 手动导出/导入 | 用户自己导文件再导入 | 零成本、零后端、最隐私 | 不自动、体验差（穷人同步）|
| ② 传统云账号 | 数据进我们数据库，各端拉取 | 好实现、体验顺 | 数据在我们服务器（与本地优先有张力）、要运维 |
| ③ **本地优先 + 同步层**（推荐）| 本地为主，后台静默同步，可端到端加密 | 离线可用 + 自动 + 隐私，最贴产品魂 | 实现最复杂、需选对引擎 |

**推荐 ③，分阶段落地，前期优先用现成同步引擎而非纯自建。**

### 3.2 冲突解决
- 语迹是「**个人多设备**」而非「多人协作」→ 用**字段级「最后写入胜出」(LWW)** + 逻辑时钟即可，
  不必上重型 CRDT（多人协作才更需要）。
- 极少数真冲突：**保留两版让用户选**，不静默覆盖（呼应抹平摩擦/不惩罚）。

### 3.3 离线与合并
- 离线编辑进**本地待同步队列**；重连后按 `updatedAt` 合并、推送 tombstone。
- 同步是后台静默的；失败重试，绝不阻塞用户操作。

### 3.4 技术候选（到时挑一个）
Dexie Cloud / Replicache(Zero) / PowerSync / ElectricSQL / RxDB —— 均为「IndexedDB 或 SQLite 本地 + 自动同步」的成熟方案，可省掉大量自建同步的坑。也可用 Turso/SQLite embedded replica 自建。

### 3.5 隐私（可选增强）
客户端**端到端加密**后再上云，服务器零知识（只存看不懂的密文）——「数据是用户的」的最强保证。成本与复杂度更高，作为 P2。

## 4. 身份认证设计（微信登录 / 手机号登录）

> 定位：登录用于**绑定多端身份、解锁同步与恢复**；不登录仍可本地完整使用。
> 起步阶段甚至可先做「**设备配对码 / 扫码**」在两台设备间直连同步——**无需账号**，最贴隐私与低摩擦；账号体系随后叠加。

### 4.1 微信登录
按入口不同，三种标准方式（可都支持，用 **unionid 统一身份**）：

| 场景 | 方案 | 关键点 |
|---|---|---|
| 桌面网页 | 微信开放平台「网站应用」**扫码登录**（OAuth2 `snsapi_login`）| 前端拿 `code` → 后端用 appid+secret 换 `access_token` + `openid`/`unionid` |
| 微信内 H5 / 公众号 | 公众号网页授权（`snsapi_userinfo`）| 在微信内置浏览器静默/弹窗授权 |
| 小程序 | `wx.login` → `code2session` | 拿 `openid`/`unionid`，体验最顺 |

- **unionid 是跨端统一的关键**：同一微信开放平台主体下的网站应用 + 小程序 + 公众号**共享 unionid**，
  用它做账号主键，用户在任意端登录都是同一身份，天然支撑多端同步。
- **硬性前置条件**：微信开放平台账号（**企业资质**、有年审费）、**ICP 备案域名**、HTTPS、必须有**后端**藏 `appsecret` 并完成换 token（前端不能直接调）。
- 个人开发者/无企业资质时，微信网站应用申请不下来 → 届时**手机号登录作为主路径**。

### 4.2 手机号登录
| 方案 | 机制 | 取舍 |
|---|---|---|
| 短信验证码（OTP）| 第三方短信（阿里云/腾讯云 SMS）后端发码、校验 | 通用、门槛低；有短信成本、需防刷 |
| 一键登录（号码认证）| 运营商本机号码校验（阿里云号码认证 / 极光等）| 体验最好（免输验证码）；需资质、按次计费 |

- 手机号作为账号主键，可与微信 `unionid` **互相绑定**（一个账号多种登录方式）。
- 防滥用：验证码频率限制、图形/滑块验证、IP/设备风控。

### 4.3 账号与本地数据的衔接
- 匿名期：数据带本地 `deviceId`，完全可用。
- 首次登录：把本地匿名数据**归并**到该账号（首端直接归属；后续端走同步合并）。
- 退出登录：本地数据保留可用、可导出（不锁定）。

## 5. 真实语音识别（相关缺口，单列）

> 现状：原型只实现 Web Speech **兜底**；Chrome 的 Web Speech 依赖 Google 后端，**国内不可用 → 回退示例**。
> 对真实用户（尤其国内），语音**目前等于用不了**。这是「从原型到产品」的核心缺口。

产品化方案（需后端中转藏 key）：
- **微信小程序录音 + ASR**（若走小程序端）、**科大讯飞**、**腾讯云/阿里云 ASR**，或自建 **Whisper**。
- 与「日程解析」配合：ASR 出文本 → （Phase 2）LLM 拆多意图 → 待执行清单 → 用户确认执行。
- 兜底链：真实 ASR 失败 → 文本输入；解析失败 → 逐句规则解析。永不崩、永不静默。

## 6. 落地路线图（每步可独立上线）

- **P0 · 地基**：统一 Store + IndexedDB；**两端打通同一份数据**；完善导出/导入。
  （做完即解决「同浏览器内手机/网页不互通」）
- **P1 · 账号 + 云同步**：手机号登录（先）/ 微信登录（资质就绪后）；单账号同步（LWW + tombstone 复用回收站）+ 离线队列。
- **P2 · 隐私增强**：端到端加密、设备配对、温和的冲突选择 UI。
- 并行 · **真实 ASR 接入**（见第 5 节），让语音对真实用户真正可用。

## 7. 待你拍板的决策点
1. 同步是否核心卖点？（决定投入力度）
2. 隐私强度：要不要端到端零知识加密（更强但更贵更复杂）？
3. 登录形态优先级：手机号优先？微信优先（需企业资质 + 备案域名）？还是先做无账号设备配对？
4. 平台目标：纯网页/PWA？微信小程序？真原生 App？（直接影响存储、同步、微信登录的实现路径）
5. 自建 vs 现成同步引擎（控制力 vs 速度/依赖）。
6. 真实 ASR 选型：讯飞 / 腾讯云 / 阿里云 / 小程序录音 / 自建 Whisper。
